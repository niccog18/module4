Learning Objectives

By the end of this lesson, you will be able to:

    Identify resources, relationships, and actions for any application domain
    Design a complete REST API endpoint structure on paper before writing code
    Define request and response schemas for each endpoint
    Apply REST conventions consistently across an entire API design

There's a reason architects draw blueprints before construction workers pour concrete. Designing before building lets you catch problems when they're cheap to fix, on paper, rather than after you've written 500 lines of code.

This lesson is about API design as a thinking exercise. You won't write a single line of Python. Instead, you'll learn to map a real-world problem into a REST API structure. This is exactly what professional API developers do before opening their code editor, and it's what you'll do for Part 2 of this module's project.


Step 1: Identify Your Resources
----------------------------------
Start by asking: what are the things in this system? List the nouns.

Let's design an API for a study group platform, think of a simplified version of Slack for study groups.

The things in this system:

    Users - people who use the platform
    Groups - study groups that users join
    Messages - messages posted within a group
    Assignments - homework or tasks within a group

These are your resources. Each one gets its own set of endpoints.


Step 2: Define the Relationships
----------------------------------
Resources don't exist in isolation. Map out how they connect:

    A user can belong to many groups (many-to-many)
    A group has many messages (one-to-many)
    A group has many assignments (one-to-many)
    A message belongs to one user and one group
    An assignment belongs to one group

These relationships determine your nested resource paths:
/groups/{id}/messages     ← messages in a group
/groups/{id}/assignments  ← assignments in a group
/groups/{id}/members      ← users in a group


Step 3: Design the Endpoints
----------------------------------
For each resource, define the CRUD operations:
USERS
-----
POST   /users                    ← Register a new user
GET    /users/{id}               ← Get user profile
PUT    /users/{id}               ← Update user profile
DELETE /users/{id}               ← Delete user account

GROUPS
------
POST   /groups                   ← Create a new group
GET    /groups                   ← List all groups
GET    /groups?subject=python    ← Filter groups by subject
GET    /groups/{id}              ← Get group details
PUT    /groups/{id}              ← Update group info
DELETE /groups/{id}              ← Delete a group

GROUP MEMBERSHIP
----------------
POST   /groups/{id}/members      ← Join a group (add yourself)
GET    /groups/{id}/members      ← List group members
DELETE /groups/{id}/members/{user_id}  ← Leave a group

MESSAGES
--------
POST   /groups/{id}/messages     ← Post a message to a group
GET    /groups/{id}/messages     ← Get messages in a group
GET    /groups/{id}/messages?limit=50&before=2024-01-15  ← Paginated messages
DELETE /groups/{id}/messages/{msg_id}  ← Delete a message

ASSIGNMENTS
-----------
POST   /groups/{id}/assignments  ← Create an assignment
GET    /groups/{id}/assignments  ← List assignments
GET    /groups/{id}/assignments/{assign_id}  ← Get assignment details
PUT    /groups/{id}/assignments/{assign_id}  ← Update an assignment
DELETE /groups/{id}/assignments/{assign_id}  ← Delete an assignment


Step 4: Define the Schemas
----------------------------------
For each endpoint that accepts or returns data, define the shape:

POST /users - Request Body:
{
  "name": "string (required)",
  "email": "string (required, unique)",
  "password": "string (required, min 8 characters)"
}

POST /users - Response (201 Created):
{
  "id": "integer",
  "name": "string",
  "email": "string",
  "created_at": "datetime"
}

Note: the password is never in the response. The id and created_at are generated by the server.

POST /groups/{id}/messages - Request Body:
{
  "content": "string (required, max 2000 characters)"
}

GET /groups/{id}/messages - Response (200 OK):
{
  "data": [
    {
      "id": "integer",
      "content": "string",
      "author": {
        "id": "integer",
        "name": "string"
      },
      "created_at": "datetime"
    }
  ],
  "total": "integer",
  "limit": "integer",
  "offset": "integer"
}


Step 5: Define Authentication and Authorization
----------------------------------
Decide which endpoints need authentication and who can access what:

Endpoint
	1. GET /users/{id}
    2. POST /groups
    3. POST /groups/{id}/messages
Auth Required?
	1. Yes 
    2. Yes 
    3. Yes 
Who Can Access?
	1. Any logged-in user 
    2. Any logged-in user 
    3. Only group members 
POST /users
	1. PUT /users/{id}
    2. DELETE /groups/{id}
    3. GET /groups/{id}/messages
No
	1. Yes
    2. Yes
    3. Yes
Anyone (registration)
    1. Only the user themselves
    2. Only the group creator
    3. Only group members

This matrix tells you exactly where to add authentication checks in your code, before you write any of it.


Step 6: Plan Error Responses
----------------------------------
Think about what can go wrong at each endpoint:

POST /groups/{id}/messages:

    201: Message created successfully
    400: Message content is empty or too long
    401: User is not authenticated
    403: User is not a member of this group
    404: Group does not exist

Planning error responses forces you to think about edge cases before they become bugs.
The Design Document Template

Bring it all together in a structured document:
API Design: Study Group Platform
Version: v1
Base URL: <https://api.studygroups.com/v1>

Authentication: JWT Bearer token
Rate Limit: 100 requests per minute

Resources:
  1. Users
  2. Groups
  3. Messages
  4. Assignments

Endpoints:
  [List each endpoint with method, path, description,
   request schema, response schema, and error codes]

Relationships:
  [Describe how resources connect]

Notes:
  [Any design decisions or trade-offs worth documenting]

This is a professional-quality deliverable. Teams use documents like this to align before building, to onboard new developers, and to communicate with frontend teams who need to know what the API will look like.
