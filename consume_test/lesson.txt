So far you've been making API calls from Python scripts using the requests library. That's one tool for the job, and it's the one you'll use most for building applications. But there are two other tools every API developer should know: Postman and curl.

Think of it like having different tools for different tasks. A hammer, a screwdriver, and a wrench can all tighten a bolt, but each is better suited to certain situations. Python requests is for building. Postman is for exploring and testing. curl is for quick checks and automation.
Postman: The API Workbench

Postman is a desktop application (and web app) designed specifically for working with APIs. It gives you a visual interface for building requests, inspecting responses, and organizing your API calls into collections.

Why Postman matters:

    Visual interface - you see headers, body, params, and response all at once
    Collections - save and organize your API calls for reuse
    Environments - switch between development, staging, and production URLs with one click
    History - see every request you've made
    Team collaboration - share collections with colleagues
    Testing - write automated tests that verify response data

When to use Postman:

    Exploring a new API for the first time
    Testing your own API endpoints during development
    Documenting API calls for your team
    Debugging an API that isn't behaving as expected

Postman Basics

When you open Postman, you'll see:

    Method dropdown - Choose GET, POST, PUT, PATCH, DELETE
    URL bar - Enter the endpoint URL
    Tabs below the URL - Params, Authorization, Headers, Body
    Send button - Fire the request
    Response panel - Shows status code, time, size, headers, and body

The Params tab lets you add query parameters visually, no need to type ?key=value in the URL. Postman builds the URL for you.

The Body tab (for POST/PUT/PATCH) lets you choose the format: raw JSON, form data, or file upload. Select "raw" and "JSON" for REST APIs.
Collections

A Postman collection is a folder of saved API requests. You can organize them by API, by feature, or by workflow:
My API Collection
├── Users
│   ├── GET All Users
│   ├── GET User by ID
│   ├── POST Create User
│   └── DELETE User
├── Posts
│   ├── GET All Posts
│   ├── GET Posts by User
│   └── POST Create Post
└── Auth
    ├── POST Login
    └── POST Register

Collections are shareable, you can export them as JSON files and import them on another machine.
curl: The Command-Line Tool

curl (pronounced "curl", it stands for "client URL") is a command-line tool for making HTTP requests. It comes pre-installed on macOS and Linux, and is available for Windows.

Why curl matters:

    Available everywhere - any server, any terminal, no installation
    Scriptable - embed API calls in bash scripts
    Lightweight - no GUI, no overhead
    Documentation examples - many APIs show curl examples in their docs
    SSH-friendly - test APIs from remote servers where you can't install Postman

Basic curl syntax:
# GET request
curl <https://jsonplaceholder.typicode.com/posts/1>

# GET request with pretty output
curl -s <https://jsonplaceholder.typicode.com/posts/1> | python -m json.tool

# POST request with JSON body
curl -X POST <https://jsonplaceholder.typicode.com/posts> \\
  -H "Content-Type: application/json" \\
  -d '{"title": "Test", "body": "Hello", "userId": 1}'

# PUT request
curl -X PUT <https://jsonplaceholder.typicode.com/posts/1> \\
  -H "Content-Type: application/json" \\
  -d '{"title": "Updated", "body": "New body", "userId": 1}'

# DELETE request
curl -X DELETE <https://jsonplaceholder.typicode.com/posts/1>

# Request with authentication header
curl -H "Authorization: Bearer my-token" \\
  <https://api.example.com/protected>

# Show response headers
curl -I <https://jsonplaceholder.typicode.com/posts/1>

# Show both request and response headers (verbose)
curl -v <https://jsonplaceholder.typicode.com/posts/1>

curl flags to know:

    -X METHOD - specify the HTTP method
    -H "Header: value" - add a header
    -d 'data' - send data in the body
    -s - silent mode (suppress progress bar)
    -I - show only response headers
    -v - verbose mode (show everything)
    -o filename - save response to a file
