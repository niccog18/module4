Learning Objectives

By the end of this lesson, you will be able to:

    Trace the complete journey of an HTTP request from client to server and back
    Identify the four components of an HTTP request (method, URI, headers, body)
    Identify the three components of an HTTP response (status code, headers, body)
    Explain what happens at each stage of the lifecycle

You've learned the pieces, methods, status codes, URIs. Now let's see how they fit together in a single, complete transaction. Every time your code calls an API, a precise sequence of events unfolds. Understanding this sequence is the difference between blindly copying code and knowing what's actually happening.

Think of mailing a letter. You write the letter (the body), put it in an envelope (the request), write the address (the URI), add a stamp and return address (the headers), and specify whether you're sending it standard or express (the method). The postal service routes it to the destination. The recipient reads it, writes a reply, and sends it back the same way.

An HTTP request-response cycle works the same way, but it takes milliseconds instead of days.
The Request: What You Send

Every HTTP request has up to four parts:
1. Method

The HTTP verb: GET, POST, PUT, PATCH, DELETE. Tells the server what action you want.

2. URI (URL)

The address of the resource: https://api.example.com/users/42

This breaks down into:

    Protocol: https:// (secure HTTP)
    Host: api.example.com (which server)
    Path: /users/42 (which resource)
    Query parameters (optional): ?fields=name,email (modifiers)

3. Headers

Metadata about the request. Not the data itself, information about the data. Like the "From" address and "Priority" marking on an envelope.
Content-Type: application/json       ← "I'm sending JSON data"
Authorization: Bearer eyJhbGci...    ← "Here's my identity token"
Accept: application/json             ← "Please send JSON back"
User-Agent: python-requests/2.31.0   ← "I'm a Python script"

4. Body (optional)

The actual data you're sending. Present in POST, PUT, and PATCH requests. Absent in GET and DELETE (you're not sending data, you're requesting or removing it).
{
  "name": "Alice Chen",
  "email": "alice@example.com"
}

The Response: What You Get Back

Every HTTP response has three parts:
1. Status Code

The three-digit number you learned about: 200, 201, 404, 500, etc.
2. Headers

Metadata about the response:
Content-Type: application/json      ← "I'm sending JSON"
Content-Length: 256                  ← "The body is 256 bytes"
X-RateLimit-Remaining: 97           ← "You have 97 requests left"

3. Body

The actual data the server is sending back. Usually JSON for REST APIs:
{
  "id": 42,
  "name": "Alice Chen",
  "email": "alice@example.com",
  "created_at": "2024-01-15T10:30:00Z"
}

The Complete Lifecycle

Here's what happens when your Python script calls requests.get("<https://api.example.com/users/42>"):

    DNS Resolution - Your computer looks up api.example.com and gets an IP address (like looking up a phone number in a directory).
    Connection - Your computer establishes a TCP connection to the server at that IP address. If it's HTTPS, a secure encryption handshake happens here.
    Request Sent - Your HTTP request (method + URI + headers + body) is sent over the connection.
    Server Processing - The server receives the request, routes it to the correct handler, processes it (reads from a database, runs a model, etc.), and constructs a response.
    Response Sent - The server sends back the response (status code + headers + body).
    Connection Close - The connection may be closed or kept alive for reuse.
    Client Processing - Your code receives the response and processes it (response.json(), checking response.status_code, etc.).

Steps 1–6 happen in the time it takes you to blink. The requests library handles all of this automatically, you just see the final response.

Why This Matters

When an API call fails, understanding the lifecycle tells you where it failed:

    DNS failure - the domain doesn't exist (check the URL)
    Connection failure - server is down or firewall is blocking you
    Timeout - server is too slow to respond
    4xx response - your request reached the server but was rejected
    5xx response - your request reached the server but it broke

Each failure point has a different fix. Understanding the lifecycle helps you diagnose problems faster.
